## Parte A - Revisión de tu backlog de historias de usuario

### Paso 1 - Recupera tu entregable del ejercicio pre-S4

Vuelve a abrir el prompt que redactaste y las historias de usuario que generaste a partir del PRD de FlowSync en el ejercicio previo a la S4.

### Paso 2 - Contrástalo con lo que sabes ahora

Léelo con la perspectiva que tienes hoy —después del directo de S4 y del contenido de S5— y responde por escrito:

* ¿Siguen teniendo sentido las historias tal como las generaste? ¿El alcance sigue ceñido al MVP del PRD, o se coló alguna que la IA "inventó" fuera de scope?

* ¿Hay historias cuyos criterios de aceptación ahora ves incompletos o poco verificables?

* ¿Hay historias que han cambiado de naturaleza desde entonces? (porque descubriste una dependencia, porque la spec evolucionó, porque entiendes mejor el dominio).

* ¿Hay historias nuevas que no aparecieron cuando lo generaste y que ahora sí deberían estar?

* Al contrastar con el backlog que el mentor construyó en el directo de S4 sobre Linear: ¿qué priorizaste distinto tú? ¿Quién acertó y por qué?

### Paso 3 - Anota los ajustes

Cierra la Parte A con una lista breve de 2 a 4 ajustes concretos que harías a tu backlog si lo rehicieras hoy. Para cada ajuste:

```text
Ajuste: [qué cambiarías]
Motivo: [por qué — qué has aprendido o descubierto desde la S4]
```

No tienes que aplicar los ajustes. Solo dejarlos identificados como evidencia de que un plan es un documento vivo, no un compromiso rígido.

## Parte B - Auditoría de documentación del proyecto

### Paso 4 - Decide el formato del artefacto de auditoría

Tienes que llegar a la sesión con un artefacto de auditoría consultable, no con notas sueltas o impresiones generales. La forma la decides tú. Opciones válidas:

* Tabla en Markdown con columnas (tipo de documentación, estado, observación, ubicación).

* Informe estructurado por secciones.

* Matriz visual.

* Issue o ticket en tu herramienta habitual con la auditoría como cuerpo.

* Cualquier otro formato que cumpla los requisitos siguientes.

Lo que sí debe cumplir el formato que elijas:

* Ser un artefacto consultable, no impresiones de cabeza.

* Cubrir todos los tipos de documentación del listado del Paso 5.

* Para cada tipo, indicar el estado y una observación específica.

* Cerrar con un Top 3 de carencias y un Top 3 de fortalezas.

### 🛠 Andamiaje opcional con IA (si te ayuda a arrancar)

Tienes disponible un prompt asistente que monta el proyecto y te genera la plantilla vacía del artefacto de auditoría, y luego te acompaña tipo por tipo con preguntas guía. Úsalo como andamiaje, no como auditor: la evaluación de cada tipo, el estado y la observación la haces tú. La IA no sabe si tu README permite arrancar el proyecto desde cero ni si una spec diverge del código; eso lo compruebas tú mirando el repo. Si no lo necesitas, ve directo al repositorio: es perfectamente válido y a menudo más rápido para un perfil senior.

### Paso 5 - Audita cada tipo de documentación

Recorre tu copia del repositorio y evalúa el estado de cada tipo de documentación. Para cada uno, asigna un estado:

* Completa: existe, está actualizada y cumple su función.

* Parcial o pobre: existe pero está incompleta, desactualizada, mal organizada o no cumple su función.

* Inexistente: no hay nada que cubra esto.

Y añade una observación específica (una o dos líneas): qué falta, qué chirría, qué está bien, o dónde se encuentra si está disperso.

Tipos a auditar:

* [README de proyecto. ¿Permite que alguien nuevo arranque el proyecto desde cero (instalar dependencias, configurar .env, correr migraciones, levantar el servidor) sin tener que preguntar?

* Descripción de la arquitectura general. ¿Hay un documento que explique cómo está organizado el sistema, qué componentes lo forman y cómo se relacionan?

* Documentación de la API o de los endpoints. El proyecto expone una API (auth, /health, etc.). ¿Está documentada de forma que otro pueda integrarse contra ella sin leer el código de las rutas?

* Docstrings y comentarios significativos en código (TSDoc/JSDoc). ¿Los controllers, services y modelos relevantes tienen documentación inline útil, o solo comentarios genéricos (o ninguno)?

* Decisiones técnicas registradas. ¿Existen ADRs (Architecture Decision Records) o equivalentes que justifiquen las decisiones tomadas y por qué se descartaron otras alternativas?

* Guía operacional. ¿Hay instrucciones para desplegar el proyecto, runbooks para incidencias típicas, o documentación de troubleshooting?

* Convenciones de código del proyecto. ¿Hay un documento (o un CLAUDE.md / AGENTS.md) que explique cómo se nombra, cómo se estructura, qué patrones se siguen?

* Especificación OpenSpec y su trazabilidad con el código. ¿La especificación está al día respecto al código actual? ¿Se ven divergencias entre lo que dice la spec y lo que hace el código? (Pista: mira si hay endpoints en la spec que no existen en el código, o al revés).

Si detectas tipos adicionales de documentación que no encajan en esta lista, añádelos al final. Si alguno claramente no aplica, márcalo como "no aplica" y justifica brevemente por qué.

### Paso 6 — Cierra la auditoría con dos Top 3

Después de la auditoría detallada, cierra el artefacto con dos listas cortas:

```text
Top 3 carencias que más duelen:
1. [Qué falta o está mal, y por qué es importante]
2. ...
3. ...

Top 3 cosas que ya están bien:
1. [Qué funciona, para no romperlo en la sesión]
2. ...
3. ...
```

El Top 3 de carencias define dónde ponemos el foco en el directo. El Top 3 de fortalezas evita que la sesión rehaga cosas que ya funcionan.

### Paso 7 - Exploración rápida de tres formatos de documentación

Para llegar al directo con vocabulario básico, dedica un máximo de 5 minutos a cada uno de estos tres formatos. No tienes que dominarlos, solo familiarizarte con su pinta.

* Un ejemplo de diagrama C4 (busca "C4 model example" en imágenes, o revisa c4model.com).

* Un ejemplo de ADR (consulta el repositorio público joelparkerhenderson/architecture-decision-record).

* Una especificación OpenAPI pública (la de GitHub, Stripe, o cualquier servicio que tengas a mano).

Para cada uno, anota una sola línea: ¿qué aporta a primera vista que no aporten unas notas sueltas en Markdown?

No leas más de lo necesario para responder esa línea. La profundidad la trabajamos en el directo.

## Pasos para entregar mediante Pull Request

Haz un fork del repositorio usando el botón ubicado arriba a la derecha.

Clona tu fork en tu computadora. Será un proyecto con el mismo nombre, pero bajo tu usuario.

Completa el ejercicio:

* Rellena el prompt.

* Agrega o modifica los archivos correspondientes dentro de tu carpeta.

* Sube el documento único con la evidencia, la Parte A y las observaciones.

Crea una nueva rama para tu solución, por ejemplo:

```bash
git checkout -b solved-stopwatch
```

Haz commit de tus cambios:

```bash
git add .
git commit -m "Solve stopwatch exercise"
```

Sube tu rama al repositorio:

```bash
git push origin solved-stopwatch
```

En la interfaz de GitHub de tu fork aparecerá un aviso para crear el Pull Request.

Crea el Pull Request hacia el repositorio original.

Esa será tu entrega final.

## Recursos recomendados

Para la exploración de formatos del Paso 7:

* C4 model: página oficial https://c4model.com. La portada con los cuatro niveles (Context, Containers, Components, Code) es suficiente.

* ADRs: repositorio público joelparkerhenderson/architecture-decision-record en GitHub. Plantillas y ejemplos reales.

* OpenAPI: especificación pública de GitHub (https://docs.github.com/en/rest) o de Stripe (https://docs.stripe.com/api). Cualquiera sirve.

Lecturas complementarias (opcionales):

* "Docs as Code" — https://www.writethedocs.org/guide/docs-as-code/. Introducción corta al concepto.

* "Living Documentation" de Cyrille Martraire: por qué la documentación debe vivir con el código y no en un Word aparte.
