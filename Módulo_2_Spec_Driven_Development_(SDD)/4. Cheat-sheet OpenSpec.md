## Slash commands

### Perfil core (default — los 3 que vas a usar el 90% del tiempo)

| Comando                       | Para qué sirve                                                               |
| ----------------------------- | ---------------------------------------------------------------------------- |
| `/opsx:propose <descripción>` | Crea un change completo en un solo paso (proposal + specs + design + tasks). |
| `/opsx:apply`                 | Implementa todas las tasks del change activo.                                |
| `/opsx:archive`               | Archiva el change y fusiona delta specs en specs vivas.                      |

### Perfil expanded (opt-in — más control granular)

| Comando                | Para qué sirve                                                       |
| ---------------------- | -------------------------------------------------------------------- |
| `/opsx:new <nombre>`   | Crea el directorio del change y solo el primer artefacto.            |
| `/opsx:continue`       | Crea el siguiente artefacto del change actual.                       |
| `/opsx:ff <nombre>`    | Fast-forward: crea todos los artefactos de planificación de una vez. |
| `/opsx:explore <tema>` | Modo exploración: piensa sin comprometerse a un change.              |
| `/opsx:verify`         | Verifica que la implementación cumple las specs.                     |
| `/opsx:onboard`        | Walkthrough interactivo (~15 min) para nuevos usuarios.              |
| `/opsx:bulk-archive`   | Archiva múltiples changes completados en una sola operación.         |

## 💻 CLI (terminal)

```bash
openspec --version              # Versión instalada
openspec init                   # Inicializa OpenSpec en el proyecto actual
openspec config profile         # Cambia perfil core ↔ expanded
openspec config list            # Lista configuración actual
openspec update                 # Regenera slash commands después de cambios

openspec list                   # Lista changes activos
openspec show <change-name>     # Muestra detalles de un change
openspec validate <change-name> # Valida formato del spec
openspec view                   # Dashboard interactivo (TUI)
openspec feedback               # Crea un GitHub Issue con metadata
```

## 📁 Estructura de carpetas tras openspec init

```text
openspec/
├── project.md                  # contexto del proyecto (lo curás vos)
├── changes/                    # propuestas activas
│   └── <change-name>/
│       ├── proposal.md         # por qué + qué
│       ├── specs/              # delta specs por capability
│       │   └── <capability>/
│       │       └── spec.md
│       ├── design.md           # decisiones técnicas
│       └── tasks.md            # checklist de implementación
└── specs/                      # specs vivas (se llenan al archivar)
    └── <capability>/
        └── spec.md

# Tras `/opsx:archive`, el change se mueve a:
openspec/changes/archive/<YYYY-MM-DD>-<change-name>/
```

## 📝 Sintaxis delta spec (copia esto)

```md
# Delta for <capability>

## ADDED Requirements

### Requirement: <Nombre del requirement>
The system MUST <comportamiento esperado>.

#### Scenario: <nombre escenario>
- WHEN <acción que ocurre>
- THEN <resultado esperado>
- AND <resultado adicional>  (opcional)

## MODIFIED Requirements

### Requirement: <Nombre del requirement existente>
The system SHALL <nuevo comportamiento>.
(Previously: <comportamiento previo>)

#### Scenario: <nombre>
- WHEN ...
- THEN ...

## REMOVED Requirements

### Requirement: <Nombre del requirement> (Deprecated en favor de Y)
```

## Errores comunes y cómo resolverlos

| Síntoma                                                   | Causa probable                             | Solución                                                         |
| --------------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------- |
| Los slash commands no aparecen en Claude Code             | Claude Code no detectó `.claude/skills/`   | Reiniciar Claude Code completamente (no `/clear`)                |
| `openspec init` dice "not initialized" tras inicializar   | Bug de pre-flight en versiones antiguas    | Actualizar OpenSpec a versión reciente                           |
| Comandos `/openspec:*` en lugar de `/opsx:*`              | Proyecto creado antes de OpenSpec 1.0      | Correr `openspec init` de nuevo; estructura compatible           |
| Delta spec rechazada con "validation error"               | Sintaxis ADDED/MODIFIED/REMOVED incorrecta | `openspec validate <change>` para ver el error específico        |
| Implementación se desvía de la spec                       | Contexto contaminado                       | `/clear` antes de `/opsx:apply`; la IA releerá la spec del disco |
| Dos devs hacen propose sobre la misma capability a la vez | Concurrent changes                         | Coordinar como con merges de Git; archivar en orden              |

## Flujo de trabajo típico (memorízalo)

```text
┌─────────────────────────────────────────────────────────────┐
│  1.  /opsx:propose "describe lo que queres hacer"           │
│      → AI genera proposal.md + specs/ + design.md + tasks.md│
│                                                             │
│  2.  REVISÁS los 4 archivos. Editás si la IA improvisó.     │
│      → Este es el momento más valioso del flujo.            │
│                                                             │
│  3.  /opsx:apply                                            │
│      → AI implementa, escribe tests, ejecuta tests.         │
│                                                             │
│  4.  Probás manualmente. Si todo OK:                        │
│                                                             │
│  5.  /opsx:archive                                          │
│      → Delta spec se fusiona con la spec viva.              │
│      → Change se mueve a archive/.                          │
│                                                             │
│  6.  Commit + push. PR a main.                              │
└─────────────────────────────────────────────────────────────┘
```
