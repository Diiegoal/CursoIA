## La pregunta correcta no es «¿qué herramienta uso?»

La pregunta correcta es:

> **¿Qué tipo de tarea voy a realizar y qué nivel de control humano necesito?**

La herramienta aparece como consecuencia de esa respuesta.

Si haces esto al revés —eliges primero la herramienta y después buscas tareas para utilizarla— ocurren dos cosas predecibles:

1. Utilizas mal la herramienta.
2. Atribuyes el mal resultado a la inteligencia artificial.

Esta lección te proporciona el modelo mental necesario para invertir ese orden.

# Taxonomía de 2026: cuatro categorías según el modo de uso

Más allá del modelo subyacente, los copilotos se diferencian por:

* Dónde viven.
* Cómo se interactúa con ellos.
* Cuál es su unidad de trabajo.
* Cuánto control conserva el desarrollador.
* Cuánto tiempo pueden trabajar antes de devolver un resultado.

Estas son las cuatro categorías canónicas en 2026.

---

## Categoría A — Integrados en el IDE

### Visuales y con diferencias en línea

Viven dentro de tu editor, por ejemplo:

* Visual Studio Code.
* JetBrains.
* Xcode.
* Neovim.
* Zed.

Su unidad de trabajo es el **cursor**.

Permiten realizar tareas como:

* Autocompletados en línea.
* Ediciones de múltiples archivos mediante un prompt en un chat lateral.
* Refactorizaciones con diferencias visuales.
* Aceptar o rechazar cambios directamente desde el editor.

### Representantes

* Cursor.
* GitHub Copilot.
* Windsurf.
* Cline.
* Continue.dev.
* Zed AI.
* JetBrains AI Assistant.

### Modo de interacción

> **Edit-as-you-go:** editar a medida que avanzas.

El flujo habitual es:

1. Tú escribes.
2. El modelo sugiere.
3. Tú envías un prompt.
4. El modelo edita.
5. El modelo muestra las diferencias.
6. Tú aceptas o rechazas los cambios.

### Latencia tolerable

* Menos de **500 milisegundos** para el autocompletado.
* Algunos segundos para las respuestas del chat.

### Mejor para

* Trabajo iterativo dentro de un archivo.
* Funcionalidades pequeñas.
* Exploración guiada.
* Refactorizaciones visuales.
* Cambios que necesitan revisión inmediata.

---

## Categoría B — Agentes de terminal o CLI

Viven en la terminal.

Su unidad de trabajo es la **tarea**.

Les describes un objetivo y el agente puede:

1. Analizar la solicitud.
2. Elaborar un plan.
3. Leer archivos.
4. Buscar código.
5. Modificar archivos.
6. Ejecutar comandos.
7. Ejecutar compilaciones y pruebas.
8. Regresar con los resultados.

### Herramientas disponibles

Según el agente, puede tener herramientas como:

* `Read`
* `Edit`
* `Bash`
* `Grep`
* Búsqueda de archivos.
* Ejecución de pruebas.
* Gestión de Git.
* Integración con servidores MCP.

### Representantes

* Claude Code.
* Codex CLI de OpenAI.
* Aider.
* OpenCode.
* Gemini CLI.
* Copilot CLI, con disponibilidad general desde febrero de 2026.
* Cline CLI.

### Modo de interacción

> **Plan-then-execute:** primero planificar y después ejecutar.

El flujo habitual es:

1. Tú describes el objetivo.
2. El agente analiza el repositorio.
3. El agente propone o construye un plan.
4. El agente trabaja durante varios minutos.
5. Tú revisas el resultado.

El resultado puede presentarse como:

* Archivos modificados.
* Un commit.
* Una rama.
* Un pull request.
* Un informe con los cambios y pruebas realizadas.

### Latencia tolerable

> Varios minutos por tarea pueden ser aceptables.

### Mejor para

* Refactorizaciones de múltiples archivos.
* Creación de scripts.
* Pipelines.
* Exploración de codebases desconocidos.
* Ejecución de migraciones.
* Compilaciones.
* Pruebas.
* Trabajo paralelo mediante `tmux`.
* Trabajo paralelo mediante `git worktrees`.

---

## Categoría C — Agentes autónomos independientes en la nube

Viven en la nube.

Su unidad de trabajo es el **ticket**.

Puedes asignarles un elemento de trabajo proveniente de plataformas como:

* Jira.
* Linear.
* GitHub Issues.

El agente ejecuta el trabajo de manera autónoma y devuelve normalmente un pull request.

### Representantes

* Devin, de Cognition.
* Cursor Background Agents.
* GitHub Copilot Coding Agent.
* Jules, de Google.
* AWS Q Developer Agent.

### Modo de interacción

> **Fire-and-forget supervisado:** asignar el trabajo y revisarlo posteriormente.

El flujo habitual es:

1. Tú especificas la tarea.
2. El agente trabaja mientras realizas otras actividades.
3. El agente modifica el proyecto.
4. El agente crea un pull request.
5. Tú revisas el resultado.

### Latencia tolerable

> El trabajo puede tardar horas.

### Mejor para

* Tareas bien especificadas.
* Paralelización masiva.
* Trabajo asíncrono.
* Trabajo fuera del horario habitual.
* Tareas independientes que puedan revisarse después mediante un pull request.

---

## Categoría D — Herramientas especializadas

Estas herramientas resuelven un subproblema concreto del ciclo de vida del desarrollo de software, conocido como **SDLC**.

Pueden especializarse en:

* Revisión de código.
* Búsqueda.
* Seguridad.
* Refactorización automatizada.
* Análisis de pull requests.
* Detección de errores.

### Representantes

* Copilot Code Review, con 60 millones de revisiones acumuladas.
* Bugbot, de Cursor.
* Greptile.
* Sourcegraph Cody.
* Refact.ai.
* Sweep.dev.
* Augment.
* Tabnine.

### Modo de interacción

> **Pipeline:** se integran en el flujo de desarrollo existente.

Pueden conectarse con:

* Pull requests.
* Integración continua.
* Pipelines de despliegue.
* IDE.
* Repositorios.
* Procesos de revisión.

Normalmente operan en segundo plano.

### Mejor para

Añadir capacidades al pipeline existente sin tener que reemplazar la herramienta principal.

---

## Patrón observado entre desarrolladores senior en 2026

> 💡 El patrón predominante es utilizar un **stack híbrido**.

Una combinación habitual sería:

1. Una herramienta integrada en el IDE para autocompletado y chat:

   * Categoría A.

2. Un agente de terminal para planificación y refactorizaciones:

   * Categoría B.

3. Una herramienta especializada para revisión de código:

   * Categoría D.

Solo los casos de paralelización extrema suelen incorporar agentes autónomos en la nube:

* Categoría C.

Por tanto, no se trata necesariamente de una elección excluyente.

> Es una **arquitectura de herramientas**.

# Una diferencia que no depende de la categoría: Completion frente a Agentic

Incluso dentro de la misma categoría A puedes operar en dos modos muy diferentes:

| Dimensión         | Completion                                     | Agentic                                                          |
| ----------------- | ---------------------------------------------- | ---------------------------------------------------------------- |
| Unidad de trabajo | Línea o función                                | Tarea o funcionalidad de múltiples archivos                      |
| Iniciativa        | Reactiva: sugiere                              | Proactiva: decide qué modificar                                  |
| Control humano    | Aceptar o rechazar cada sugerencia             | Aprobar el plan y revisar el resultado                           |
| Riesgo principal  | Sugerencia incorrecta puntual                  | Cambios en cascada, degradación del código y deuda técnica       |
| Rol del prompt    | Casi nulo; el contexto principal es el código  | Crítico: objetivo, criterios y restricciones                     |
| Métrica útil      | Tasa de aceptación del autocompletado          | Aceptación del PR en el primer intento o ausencia de regresiones |
| Ejemplos          | GitHub Copilot Inline, Cursor Tab y Supermaven | Cursor Composer, Claude Code, Cursor Agent Mode y Copilot Agent  |

---

## Decisión arquitectónica clave: cuándo cambiar de modo

Una decisión importante consiste en saber cuándo utilizar **completion** y cuándo utilizar un modo **agentic**.

Como regla práctica:

### Utiliza completion y chat lateral cuando:

* La tarea cabe dentro de una función.
* La tarea afecta un único archivo.
* La tarea afecta solamente un par de archivos estrechamente relacionados.
* Quieres mantener control inmediato sobre cada cambio.

### Utiliza un agente cuando:

* La tarea afecta varias capas.
* Debes modificar el modelo, el servicio, el endpoint y las pruebas.
* Necesitas coordinar cambios entre múltiples archivos.
* Necesitas ejecutar comandos.
* Necesitas ejecutar migraciones.
* Necesitas compilar el proyecto.
* Necesitas ejecutar pruebas.

> Si la tarea requiere ejecutar comandos, el modo agentic es obligatorio.

---

## El coste de confundir los modos

Confundir estos dos modos es uno de los antipatrones más costosos.

### Utilizar agentic para algo trivial

Puede provocar:

* Sobrecarga innecesaria.
* Consumo excesivo de contexto.
* Mayor latencia.
* Más archivos analizados de los necesarios.
* Contaminación del contexto.

### Utilizar completion para una tarea de múltiples archivos

Puede provocar:

* Código inconsistente entre archivos.
* Cambios incompletos.
* Capas desalineadas.
* Pruebas que no reflejan la implementación.
* Contratos incompatibles entre componentes.

# Matriz de decisión: cinco criterios prácticos

Cuando un desarrollador senior elige una herramienta en 2026, no debería basarse únicamente en benchmarks.

Debe utilizar cinco criterios prácticos:

1. Tamaño y forma del codebase.
2. Lenguaje principal.
3. Política de privacidad y cumplimiento.
4. Presupuesto.
5. Estilo del desarrollador.

---

## Criterio 1 — Tamaño y forma del codebase

| Tipo de codebase                                           | Recomendación                                                                                   |
| ---------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| Greenfield o pequeño, con menos de 50.000 líneas de código | Casi cualquier herramienta puede funcionar; los agentes autónomos pueden destacar               |
| Mediano, entre 50.000 y 500.000 líneas de código           | Herramienta integrada en el IDE con buena indexación, como Cursor, Copilot Workspace o Windsurf |
| Monorepo grande, con más de 500.000 líneas de código       | Herramientas con subagentes y búsqueda agentic, como Claude Code o Cursor con Background Agents |
| Código distribuido entre muchos repositorios               | Cualquier agente CLI acompañado de scripts y `git worktrees` paralelos                          |

---

## Criterio 2 — Lenguaje principal

### Python, JavaScript y TypeScript

Puede utilizarse prácticamente cualquier herramienta.

La mayoría de los modelos están sobreentrenados en estos lenguajes.

### Rust, Go y programación de sistemas

Claude Code y Zed muestran un mejor manejo de tipos según investigaciones independientes.

### Java, Kotlin y .NET

Opciones destacadas:

* GitHub Copilot.
* JetBrains AI Assistant.
* Amazon Q.

Amazon Q puede ser especialmente relevante para migraciones o proyectos de `.NET` hacia AWS.

### Lenguajes de nicho

Por ejemplo:

* Elixir.
* Clojure.
* OCaml.
* Erlang.

Se recomiendan herramientas que permitan **BYOK**, es decir, utilizar una clave propia de proveedor, acompañadas de un modelo de razonamiento.

Ejemplos:

* Aider.
* Cline.
* OpenCode.
* Opus 4.7.
* GPT-5.4.

### Frontend

Para tecnologías como:

* React.
* Vue.
* Svelte.

Prácticamente cualquier herramienta puede funcionar.

También pueden utilizarse herramientas especializadas para construir interfaces rápidamente, como:

* v0.
* Lovable.

---

## Criterio 3 — Política de privacidad y cumplimiento

### Cuando el código puede salir hacia APIs públicas

Puede utilizarse prácticamente cualquier herramienta.

### Cuando el código no puede salir

Esto puede ocurrir en sectores como:

* Banca.
* Salud.
* Defensa.
* Gobierno.
* Infraestructura crítica.

Opciones posibles:

* Cline con modelos locales.
* Aider con modelos locales.

* Qwen3.
* DeepSeek.
* Tabnine en modalidad local u *on-premises*.
* Cursor Privacy Mode con Business+.
* Amazon Q dentro de una VPC de AWS.
* Claude Code mediante Amazon Bedrock.
* Claude Code mediante Google Vertex AI.

### Cuando existe auditoría obligatoria

Si cada interacción debe quedar registrada, pueden considerarse:

* GitHub Copilot Enterprise.
* Cursor Business+.
* Claude Code Team con registros de auditoría.

## Criterio 5 — Estilo del desarrollador

### Visual y orientado al trabajo en línea

Si escribes código y aceptas sugerencias a medida que avanzas:

> Utiliza una herramienta integrada en el IDE.

### Orientado a planes y refactorizaciones

Si primero piensas en la arquitectura y posteriormente ejecutas cambios en bloques:

> Utiliza un agente de terminal o CLI.

### Asíncrono y multitarea

Si prefieres asignar tareas y supervisarlas después:

> Utiliza agentes autónomos en la nube.

### Usuario avanzado de terminal

Puedes utilizar cualquiera de las categorías anteriores, excepto aquellas que funcionen exclusivamente dentro de un IDE.

# Benchmarks: cómo interpretarlos sin engañarte

Si vas a comparar herramientas mediante benchmarks públicos, debes concentrarte en estos cuatro:

| Benchmark                       | Qué mide                                           | Por qué importa                                                                       |
| ------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------- |
| SWE-Bench Verified              | 500 issues reales de Python                        | Es el estándar de la industria, pero puede estar contaminado                          |
| SWE-Bench Pro, de Scale AI SEAL | 1.865 tareas, cuatro lenguajes y sin contaminación | Funciona como correctivo; Opus 4.5 baja de 80,9 % a 45,9 % utilizando el mismo modelo |
| Aider Polyglot                  | 225 ejercicios de Exercism en seis lenguajes       | Es reproducible, económico de ejecutar y multilenguaje                                |
| Terminal-Bench 2.0              | Tareas reales dentro de una terminal               | Evalúa el harness completo y no solamente el modelo                                   |

---

## Trampa común al comparar benchmarks

> ⚠️ Comparar directamente las puntuaciones de SWE-Bench Verified publicadas por distintos proveedores.

La mayoría de estas puntuaciones:

* Son reportadas por los propios proveedores.
* Utilizan un scaffolding propio.
* Utilizan diferentes herramientas.
* Utilizan distintas configuraciones.
* Pueden utilizar diferentes cantidades de intentos.
* Pueden utilizar estrategias distintas para seleccionar resultados.

Si el proveedor A informa un resultado del **87 %** y el proveedor B informa un resultado del **85 %**, es posible que la diferencia se explique más por el harness que por el modelo.

Para comparaciones más honestas pueden consultarse:

* `vals.ai`
* SWE-rebench.
* Los leaderboards de Aider.

# Framework de decisión final

Reúne los cinco criterios y aplica el siguiente árbol de decisión.

```text
1. ¿La tarea cabe dentro de una función o un archivo?
   └── Sí → Utiliza completion.
            Herramienta integrada en el IDE, en modo Tab.

2. ¿La tarea afecta varios archivos o varias capas?
   └── Sí → Utiliza un modo agentic.

       2.1. ¿Necesitas ejecutar comandos?
            └── Sí → El modo agentic es obligatorio.

       2.2. ¿El codebase es grande o la tarea es extensa?
            └── Sí → Utiliza un agente CLI.
                     Ejemplos: Claude Code o Codex CLI.

       2.3. ¿Necesitas una refactorización visual
            o una iteración rápida?
            └── Sí → Utiliza un agente integrado en el IDE.
                     Ejemplos: Cursor Composer o Windsurf Cascade.

       2.4. ¿La tarea es asíncrona o paralelizable?
            └── Sí → Utiliza un agente autónomo en la nube.
                     Ejemplos: Devin o Background Agents.

3. ¿La herramienta cumple la política de privacidad?
   ├── No → Limita la selección a herramientas
   │        con modelos locales o ejecución dentro de una VPC.
   └── Sí → Aplica el criterio de presupuesto.

4. ¿El estilo del desarrollador encaja con la categoría?
   ├── No → Cambia de categoría, aunque otra herramienta
   │        tenga una puntuación superior en benchmarks.
   └── Sí → Continúa con la herramienta seleccionada.
```

# Antipatrones documentados en la selección de herramientas

## «Tengo Copilot porque la empresa lo paga; no necesito otra cosa»

Este razonamiento confunde:

> Disponibilidad con adecuación.

Copilot Pro puede ser excelente para:

* Completion.
* Edición.
* Sugerencias en línea.

Pero para refactorizaciones agentic extensas, otras categorías pueden estar mejor adaptadas.

---

## «Salió el modelo X.7, voy a cambiar de herramienta»

Este razonamiento confunde:

> Modelo con harness.

La mayoría de las herramientas principales probablemente ofrecerán acceso al nuevo modelo en cuestión de días.

Cambiar toda la herramienta solamente por un modelo puede hacerte perder:

* Configuraciones.
* Flujos de trabajo.
* Reglas.
* Memoria del proyecto.
* Integraciones.
* Experiencia acumulada.

---

## «Solo utilizo herramientas de código abierto por filosofía»

Esta es una elección válida.

Sin embargo, debes asumir el coste de utilizar un harness posiblemente menos pulido.

Herramientas como:

* Cline.
* Aider.
* OpenCode.

Han alcanzado un nivel importante de madurez, pero pueden exigir más trabajo manual de ingeniería de contexto.

---

## «Voy a probar todas las herramientas hasta encontrar la mejor»

Este comportamiento puede convertirse en el síndrome del evaluador eterno.

La verdadera curva de aprendizaje está principalmente en:

* El contexto.
* El prompt.
* La definición de tareas.
* La revisión.
* La verificación.

No está necesariamente en cambiar de herramienta cada mes.

---

## «Pago 200 USD por un plan Max, así que debo poder utilizarlo para todo»

Pagar más no compensa haber elegido una categoría incorrecta.

> Un plan Max utilizado dentro del modo equivocado continúa produciendo malos resultados.

La capacidad del modelo no reemplaza:

* Una buena selección de herramienta.
* Un contexto correctamente administrado.
* Una tarea bien definida.
* Una revisión humana adecuada.

## 4. Antes de cambiar de herramienta, formula esta pregunta

> **¿Tengo un problema de herramienta o un problema relacionado con la manera en que la utilizo?**

La mayoría de los llamados **“problemas con la IA”** pueden pertenecer realmente a:

* **Pilar 2:** contexto.
* **Pilar 3:** prompt.

Y no necesariamente al:

* **Pilar 1:** herramienta.
